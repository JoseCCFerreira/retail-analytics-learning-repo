1. Historia tecnica do caso

O caso Retail simula uma empresa com clientes, produtos, lojas, transacoes e itens vendidos. A arquitetura ensina a passagem de dados desde uma fonte operacional ate uma experiencia analitica consumivel.

FonteCSV sintetico criado por Python para garantir repetibilidade.
OLTPSQLite representa a base transacional, focada em detalhe e integridade.
OLAPDuckDB representa a camada analitica, focada em agregacoes e leitura rapida.
dbtOrganiza SQL em staging, marts, testes e documentacao tecnica.
MLCria modelos de regressao, classificacao e clustering sobre vendas.
StreamlitTransforma tabelas e metricas em dashboard exploravel.

2. Matriz completa: codigo, teoria, input e output

Ficheiro / pasta Teoria aplicada Input Output Como explicar
python/generate_data.py Simulacao controlada de dados de negocio. Parametros internos e regras de geracao. data/customers.csv, products.csv, stores.csv, transactions.csv, sales_items.csv. Cria uma fonte de dados repetivel para estudar pipeline sem depender de sistemas externos.
python/setup_sqlite.py + sql/sqlite/ OLTP, normalizacao, chaves e integridade relacional. CSVs gerados. data/retail_app.db. Representa a camada operacional onde dados detalhados seriam gravados no dia a dia.
python/setup_duckdb.py + sql/duckdb/ OLAP, fact table, leitura colunar e agregacoes. data/retail_app.db e CSVs. data/retail_analytics.duckdb com tabelas analiticas. Separa consumo analitico da base transacional, como num pequeno data mart.
dbt_retail_analytics/models/staging/ Staging: limpeza, casting e nomes consistentes. Tabelas brutas em DuckDB. stg_transactions, stg_sales_items. Cria uma camada limpa antes dos modelos finais.
dbt_retail_analytics/models/marts/ Modelacao dimensional para analise. Modelos staging. fct_sales, dim_products. Entrega modelos prontos para BI, ML e perguntas de negocio.
dbt_retail_analytics/models/schema.yml Data quality tests e contrato de dados. Modelos dbt. Testes not_null, unique e relationships. Mostra que a transformacao nao e so SQL: tambem ha validacao.
python/ml_retail.py Feature engineering, regressao, classificacao, clustering, PCA. data/retail_analytics.duckdb. data/ml_outputs/model_metrics.json, regression_feature_importance.csv, cluster_assignments.csv, pca_projections.csv. Transforma dados historicos em modelos e artefactos interpretaveis.
streamlit/app.py Visual analytics e storytelling operacional. DuckDB analitico. Dashboard local em browser. Fecha o ciclo: dados tecnicos viram leitura visual para decisao.
python/validate_project.py Validacao end-to-end. Codigo do repositorio. Pipeline executada, DuckDB validado, ML gerado e Streamlit compilado. Garante que o projeto continua executavel depois de alteracoes.

3. Outputs reais e interpretacao

3.1 Bases e dados gerados

  • data/retail_app.db: base OLTP SQLite.
  • data/retail_analytics.duckdb: base OLAP DuckDB.
  • data/*.csv: dados fonte para clientes, produtos, lojas, transacoes e itens.

unit_price  -> 0.5147

Leitura: o modelo percebe que o valor vendido depende quase todo de preco unitario e quantidade. Isto confirma a logica de negocio esperada.

3.4 Clustering e PCA

cluster_assignments.csv atribui cada cliente a grupos com base em gasto total, ticket medio, numero de transacoes, quantidade e desconto. pca_projections.csv reduz essas dimensoes para duas coordenadas, util para visualizar separacao entre grupos.

Frase para apresentacao: "No Retail, os outputs mostram que a pipeline fecha desde dados sinteticos ate bases, modelos dbt, metricas ML, clusters e dashboard. A regressao confirma regras comerciais; o clustering abre caminho para segmentacao."
