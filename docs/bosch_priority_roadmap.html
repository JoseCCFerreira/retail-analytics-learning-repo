Bosch Priority Roadmap

This page translates the role requirements into a practical study plan using the retail analytics repository. It shows what to learn first, where each topic lives in the project, and how to explain it in interview language.

1. PL/SQL 2. Advanced SQL 3. Data Modeling 4. Performance 5. dbt 6. Problem Solving
Why this matters

How to use this roadmap

The Bosch role is not asking only for generic SQL. It asks for a combination of Oracle-style database logic, strong SQL, data modeling, optimization thinking, and structured development.

Use this page together with the repository files so your study becomes practical instead of only theoretical.

Target outcome

You should be able to explain the business problem, the data model, the SQL logic, the dbt structure, and the Oracle-style PL/SQL routines as one coherent project.

Priority 1

PL/SQL and Oracle database logic

What you must know

  • Procedure vs function
  • SELECT ... INTO
  • Exception handling
  • Trigger basics
  • Package specification vs package body

Where it appears in the repo

  • sql/oracle_plsql_concepts/01_plsql_study_case.sql
  • docs/02_interview_prep.md
Interview language
I used this study case to practice Oracle-style database logic with procedures, functions, triggers, and packages. Even though the runtime engines here are SQLite and DuckDB, the PL/SQL layer was designed to train the Oracle development mindset required by the role.

Practice

  • Rewrite the store revenue procedure from scratch
  • Create a new function for total sales by product
  • Explain when you would use a trigger and when you would avoid it

Priority 3

Relational data modeling

Core ideas

  • Primary keys and foreign keys
  • Fact vs dimension
  • Grain definition
  • Operational vs analytical modeling

Grain of the core sales fact: one product in one transaction.

Interview language
I start by defining the business event and the grain. In this study case, the core sales event is one product within one transaction. From there, I separate descriptive context such as stores, products, and customers into dimensions.

Relevant files: sql/sqlite/01_create_tables.sql, dbt_retail_analytics/models/marts/fct_sales.sql, dbt_retail_analytics/models/marts/dim_*.sql.

Priority 4

Performance tuning mindset

  • Know why indexes matter
  • Reduce unnecessary scans
  • Apply filters as early as possible
  • Understand why execution plan thinking matters
  • Keep SQL simple enough to optimize iteratively

Relevant files: sql/sqlite/02_indexes.sql, sql/sqlite/03_business_queries.sql, sql/duckdb/01_analytical_queries.sql.

Interview language
My performance approach starts with the business need, then I validate grain, joins, filters, and indexes. I try to reduce unnecessary scans, apply filters early, and keep the SQL transparent enough to optimize iteratively.
Priority 5

Structured transformation and governance with dbt

What to know

  • Staging vs marts
  • ref() and dependencies
  • Data tests
  • Documentation and lineage

Relevant folders: dbt_retail_analytics/models/, dbt_retail_analytics/tests/, dbt_retail_analytics/macros/.

Interview language
I used dbt to turn isolated SQL logic into a structured analytics project with reusable models, explicit dependencies, testing, and documentation.
Priority 6

Problem solving, autonomy, and communication

These skills tie the whole case together. Do not present only code. Present your reasoning.

I start by clarifying the business question.
Then I define the grain.
Then I identify the required tables and joins.
Then I implement the logic step by step.
Finally, I validate correctness, performance, and maintainability.

Practice explaining one business question end to end in under two minutes.

Study order

Recommended learning path

  1. Read docs/01_context_and_theory.md
  2. Review the relational model in sql/sqlite/01_create_tables.sql
  3. Run the SQLite business queries
  4. Run the DuckDB analytical queries
  5. Run dbt: dbt seed, dbt run, dbt test
  6. Study the PL/SQL file and rewrite the routines by hand
  7. Use docs/02_interview_prep.md to rehearse answers aloud
Final checklist

What you should be able to say confidently

  • I can explain OLTP vs OLAP using this project
  • I can define the fact grain clearly
  • I can write grouped SQL and at least one window function without help
  • I can explain index basics and performance trade-offs
  • I can explain staging, marts, tests, and ref() in dbt
  • I can explain procedure vs function vs trigger vs package
  • I can present this study case as an end-to-end data platform
