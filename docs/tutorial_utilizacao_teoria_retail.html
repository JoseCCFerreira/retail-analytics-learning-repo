1. Visao Geral do que esta por tras

Este repositório simula uma empresa de retail e usa essa simulacao para ensinar arquitetura de dados, analise, governanca e Machine Learning. O valor do projeto nao esta so em "ter um dashboard". O valor esta em perceber o fluxo inteiro:

Geracao de dadosCria um dataset sintético reproduzível com clientes, lojas, produtos, transacoes e linhas de venda.
OLTPGuarda o estado operacional em SQLite, com chaves e tabelas relacionais.
OLAPConsolida dados em DuckDB para perguntas analíticas rápidas.
dbtAplica modelacao e testes na camada analítica.
MLTreina modelos de regressao, classificacao e clustering.
StreamlitMostra sinais do negocio num dashboard consumível.
Em termos mentais, pensa nisto como uma linha: gerar -> organizar -> modelar -> analisar -> explicar.

2. Como usar o projeto passo a passo

2.1 Preparar ambiente

python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt

2.2 Correr a pipeline de dados

python python/run_pipeline.py

Este comando chama internamente:

  • python/generate_data.py
  • python/setup_sqlite.py
  • python/setup_duckdb.py

2.3 Correr dbt

cd dbt_retail_analytics
dbt run
dbt test

2.4 Correr Machine Learning

python python/ml_retail.py

2.5 Abrir dashboard

streamlit run streamlit/app.py

2.6 Fazer uma validacao completa

python python/validate_project.py --include-dbt

3. Explicacao teorica do codigo

Ficheiro O que faz Porque existe
python/generate_data.py Gera CSVs sinteticos para clientes, produtos, lojas, transacoes e linhas de venda. Sem uma origem reproduzível, não conseguirias testar pipeline e ML sempre da mesma forma.
python/setup_sqlite.py Cria a base SQLite e carrega os CSVs para tabelas relacionais. Representa a camada operacional, onde o negocio seria gravado num sistema transacional.
python/setup_duckdb.py Lê a base operacional e cria a camada analítica em DuckDB. Separa armazenamento operacional de consumo analítico, que é um padrão real de arquitetura.
dbt_retail_analytics/models/ Cria modelos staging e marts com SQL versionado. Ajuda-te a governar transformações e a testar relações entre tabelas.
python/ml_retail.py Prepara features e treina modelos supervisionados e não supervisionados. Transforma a camada analítica em experiência de ciência de dados.
streamlit/app.py Lê o DuckDB e apresenta KPIs e gráficos. Fecha o ciclo entre dados técnicos e leitura de negócio.

3.1 Leitura conceptual do fluxo

Quando leres o código, tenta separar sempre origem, transformação e consumo:

  • origem: CSVs sintéticos;
  • transformação: SQLite -> DuckDB -> dbt;
  • consumo: ML outputs + Streamlit.

4. Teoria por tras das camadas

4.1 OLTP

OLTP significa Online Transaction Processing. Aqui guardas eventos pequenos, frequentes e detalhados. No teu caso: uma transacao, um item vendido, um cliente, uma loja. OLTP privilegia integridade e granularidade.

4.2 OLAP

OLAP significa Online Analytical Processing. Aqui já não pensas em "gravar uma compra". Pensas em "quanto vendi por mês", "qual categoria gera mais receita", "que clientes parecem premium". Por isso a modelação muda para ser mais rápida para agregações.

4.3 dbt

dbt transforma SQL em ativo de engenharia: modular, testável e versionado. O ganho não é só técnico. É também organizacional, porque defines uma camada de verdade mais clara.

4.4 Machine Learning

Este projeto usa três famílias de modelos:

  • Regressão: prever valores contínuos, como net_amount.
  • Classificação: prever classes, como customer_segment.
  • Clustering: encontrar grupos sem rótulo prévio.

5. Como interpretar os outputs

5.1 Base de dados

  • data/retail_app.db: estado operacional.
  • data/retail_analytics.duckdb: estado analítico.

5.2 Outputs de ML

Ficheiro O que significa Como ler
model_metrics.json Resumo numérico dos modelos. Olha para MAE, RMSE, R2 e Accuracy.
regression_feature_importance.csv Peso relativo das features na regressão. Ajuda a perceber que variáveis mais influenciam o valor líquido.
cluster_assignments.csv Cluster atribuído a cada observação. Podes estudar perfis típicos de compra por grupo.
pca_projections.csv Dados projetados em componentes principais. Serve para visualização e redução de dimensionalidade.
Um R2 muito alto num dataset sintético pode ser ótimo para aprendizagem, mas não significa automaticamente que o modelo seria tão forte em produção. Isso também é uma boa lição.

6. O que mais podes analisar

ClienteTicket médio por segmento, cidade e frequência de compra.
ProdutoElasticidade ao desconto, categorias dominantes e mix por campanha.
LojaComparação entre regiões, sazonalidade e performance relativa.
OperaçãoPagamentos, descontos, volume de itens por transação.
MLNovas features temporais, validação temporal, tuning e explicabilidade.
BIAdicionar filtros no Streamlit e drill-down por loja, categoria e mês.

6.1 Perguntas de negocio úteis

  • Que categorias reagem melhor a desconto?
  • Que regiões têm maior valor por transação?
  • Que segmentos de cliente parecem mais previsíveis?
  • Há padrões semanais claros no volume de compra?

7. Como estudar este projeto sem te perderes

  1. Corre a pipeline e confirma os ficheiros gerados.
  2. Abre o SQLite e vê as tabelas base.
  3. Abre o DuckDB e compara com a camada analítica.
  4. Lê os modelos dbt e os testes.
  5. Corre o ML e abre os outputs.
  6. Abre o Streamlit e compara o dashboard com as tabelas.
Se fizeres este percurso 2 ou 3 vezes, deixas de ver o projeto como "muitos ficheiros" e começas a vê-lo como sistema.
