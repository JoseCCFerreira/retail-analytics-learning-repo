1. Teoria de dados — o que precisas de saber antes de começar

OLTP vs OLAP

Existem dois mundos distintos em sistemas de dados. OLTP é o sistema operacional — o que responde a ações em tempo real como vendas, stocks, encomendas. OLAP é o sistema analítico — o que agrega histórico para responder a perguntas de negócio.

DimensãoOLTPOLAP
ObjetivoRegistar operações em tempo realAnalisar tendências e padrões
Motor neste projetoSQLiteDuckDB
Queries típicasINSERT / UPDATE / DELETESELECT com agregações, CTEs, janelas
Volume por operaçãoPequeno (1 registo)Grande (milhões de linhas)
EstruturaNormalizada (3NF)Desnormalizada (star schema)

Facto vs Dimensão

Uma fact table regista eventos de negócio com medidas quantitativas (valor, quantidade). Uma dimension table descreve contexto: quem, onde, o quê.

Fact: fct_sales

  • sales_item_id
  • transaction_id
  • product_id, store_id, customer_id
  • quantity, net_amount (medidas)

Dimensions: dim_products, dim_stores, dim_customers

  • product_name, category (descrição)
  • store_name, region (localização)
  • customer_name, segment (contexto)

Grain — conceito crítico

Grain: a pergunta que define o nível de detalhe de uma fact table.
Neste projeto: "1 produto dentro de 1 transação". Cada linha representa exatamente isso. Nunca agregues antes de definir o grain.

Normalização e por que a quebramos na camada analítica

No SQLite, o modelo está normalizado para evitar redundância. No DuckDB (e no dbt), unimos as tabelas num modelo mais largo porque na análise queremos responder rápido — o custo de armazenamento extra é irrelevante.

Decisões de design

  • Chaves primárias inteiras (INTEGER PRIMARY KEY) para joins simples e rápidos.
  • Foreign keys ativas no SQLite (PRAGMA foreign_keys = ON) para integridade.
  • Preço gravado em sales_items além de products — preserva o preço histórico no momento da venda.
  • Desconto calculado e gravado como campo derivado para evitar recálculo em cada query.

6. SQL — da query simples à análise avançada

Teoria: aggregations e GROUP BY

GROUP BY agrupa linhas por uma ou mais colunas. SUM, AVG, COUNT calculam métricas nessas partições. A regra fundamental: toda a coluna no SELECT que não seja agregada tem de estar no GROUP BY. 

-- Receita por loja por mês
SELECT
    strftime('%Y-%m', t.transaction_date) AS year_month,
    s.store_name,
    ROUND(SUM(si.net_amount), 2) AS revenue
FROM transactions t
JOIN sales_items si ON si.transaction_id = t.transaction_id
JOIN stores s       ON s.store_id = t.store_id
GROUP BY 1, 2
ORDER BY 1, 3 DESC;

Teoria: CTEs — Common Table Expressions

Uma CTE (WITH ... AS) é uma subquery com nome, executada antes da query principal. O benefício é legibilidade: divides um problema complexo em partes nomeadas. Não há ganho de performance por si só, mas o código torna-se muito mais fácil de rever e manter. 

WITH monthly_revenue AS (
    SELECT
        date_trunc('month', transaction_date) AS month_date,
        ROUND(SUM(net_amount), 2) AS revenue
    FROM fct_sales
    GROUP BY 1
)
SELECT month_date, revenue,
       revenue - LAG(revenue) OVER (ORDER BY month_date) AS delta
FROM monthly_revenue
ORDER BY month_date;

Teoria: Window Functions

As funções de janela calculam valores sem colapsar linhas. Perfeito para rankings, acumulados e comparações com períodos anteriores.

FunçãoO que fazUso típico
LAG(col)Valor da linha anteriorComparação com período anterior
RANK()Posição com saltos em empatesTop N com empates
ROW_NUMBER()Posição única sem empatesDeduplicação, top 1 por grupo
SUM() OVER()Acumulado ou total por partição% do total, running total
-- Top 5 produtos por categoria (DuckDB com QUALIFY)
SELECT
    p.category,
    p.product_name,
    ROUND(SUM(f.net_amount), 2) AS revenue,
    RANK() OVER (
        PARTITION BY p.category
        ORDER BY SUM(f.net_amount) DESC
    ) AS category_rank
FROM fct_sales f
JOIN products p ON p.product_id = f.product_id
GROUP BY p.category, p.product_name
QUALIFY category_rank <= 5
ORDER BY p.category, category_rank;

7. dbt — transformação com governança

Teoria: porquê o dbt existe

Antes do dbt, scripts SQL espalhados por pastas diferentes não tinham dependências explícitas, não tinham testes e não geravam documentação automaticamente. dbt resolve os 3 problemas: organiza modelos em camadas, testa qualidade de dados e gera lineage graph interativo.

Camadas do projeto dbt

Staging

Leitura das fontes com cast de tipos e rename de colunas. Sem lógica de negócio. View materializada.

Intermediate (opcional)

Joins complexos e enriquecimentos antes dos marts. Separa responsabilidades.

Marts

Modelos finais para consumo analítico. Tables materializadas. fct_sales, dim_products.

ref() — o conceito mais importante do dbt

{{ ref('stg_transactions') }} — o dbt resolve automaticamente a dependência. Se stg_transactions ainda não existir, o dbt constrói-o antes. Isto cria um grafo de dependências (DAG) que garante ordem de execução correta.

Executar dbt

cd dbt_retail_analytics
dbt run       # constrói todos os modelos
dbt test      # corre testes de qualidade (not_null, unique, relationships)
dbt docs generate && dbt docs serve  # abre documentação com lineage

Testes configurados (schema.yml)

  • not_null em todas as colunas críticas (IDs, amounts).
  • unique em sales_item_id e transaction_id.
  • relationships de fct_sales.transaction_id para stg_transactions.

8. PL/SQL — lógica de base de dados no estilo Oracle

Teoria: o que é PL/SQL

PL/SQL (Procedural Language / SQL) é a extensão procedural do SQL da Oracle. Permite escrever lógica com variáveis, condições, ciclos e tratamento de erros dentro da base de dados. Usado extensivamente em sistemas enterprise legacy.

Os 4 conceitos essenciais

ConceitoO que éQuando usar
Procedure Bloco de código que executa ações. Não retorna valor diretamente. Processar dados, carregar tabelas, tarefas de manutenção
Function Bloco que retorna um valor. Pode ser chamado em SELECT. Cálculos reutilizáveis em queries (ex.: LTV de cliente)
Trigger Código que corre automaticamente em INSERT/UPDATE/DELETE. Auditoria, validações automáticas — usar com cuidado
Package Agrupa procedures e functions numa unidade coerente. Organizar lógica relacionada, separar especificação de implementação

Exemplo da procedure deste projeto

CREATE OR REPLACE PROCEDURE prc_store_revenue (
    p_store_id      IN  NUMBER,
    p_total_revenue OUT NUMBER
) AS
BEGIN
    SELECT NVL(SUM(net_amount), 0)
    INTO   p_total_revenue
    FROM   fct_sales
    WHERE  store_id = p_store_id;
EXCEPTION
    WHEN OTHERS THEN
        p_total_revenue := 0;
END;
Raciocínio: o código PL/SQL neste projeto é conceitual — SQLite e DuckDB não executam PL/SQL nativamente. O objetivo é treinar a sintaxe e o raciocínio para ambientes Oracle reais.

Ficheiro de referência: sql/oracle_plsql_concepts/01_plsql_study_case.sql

9. App Streamlit — validação visual dos dados

Teoria: porquê visualizar os dados

Um pipeline de dados que produz resultados sem forma de os inspecionar visualmente é difícil de validar e de apresentar. O Streamlit resolve isso com Python puro — não precisas de BI tool, servidor separado ou JavaScript.

O que a app mostra

  • KPIs principais: total de transações, receita total, valor médio por linha.
  • Gráfico de receita por mês.
  • Top 10 produtos por receita com breakdown por categoria.

Executar

streamlit run streamlit/app.py

Abre automaticamente em http://localhost:8501.

Nota: a app lê diretamente o ficheiro DuckDB. O pipeline de dados tem de ter sido executado primeiro.

11. Conclusão

O que este projeto demonstra

Este estudo de caso não usa uma única ferramenta — usa cinco camadas que se complementam. Cada camada resolve um problema específico que as outras não resolvem:

  • Python resolve a geração e orquestração.
  • SQLite resolve o storage operacional com integridade relacional.
  • DuckDB resolve a analytics local com performance column-store.
  • dbt resolve a governança, testes e documentação.
  • Streamlit resolve a validação visual rápida.

O que aprendeste com este projeto

  • A diferença real entre OLTP e OLAP e quando cada um se aplica.
  • Como definir grain antes de qualquer query de facto.
  • Como escrever SQL avançado com CTEs e window functions com confiança.
  • Como o dbt resolve a fragmentação de SQL com dependências, testes e lineage.
  • Os conceitos fundamentais de PL/SQL para environments Oracle enterprise.

Como apresentar este projeto numa entrevista

Frase de síntese:
"Construí uma plataforma de analytics de retalho end-to-end onde simulo dados operacionais em SQLite, analiso com DuckDB usando CTEs e window functions, modelo com dbt com testes de qualidade e governança, e visualizo resultados com Streamlit. O projeto também inclui a camada de lógica PL/SQL Oracle para treino de ambientes enterprise."

Próximos passos para ir mais longe

  • Adicionar orquestração com Airflow ou GitHub Actions.
  • Introduzir lógica incremental no dbt (incremental materialization).
  • Expandir testes dbt com testes de negócio customizados.
  • Adicionar camada de monitorização de qualidade de dados (Great Expectations ou dbt expectations).
  • Publicar documentação dbt como site estático.
