Empacota o projeto, cria um repositório limpo no GitHub, sobe o código e valida Data, dbt, Machine Learning e dashboards. ⏱ 15 min

Segue este fluxo quando quiseres transformar a pasta atual num repositório partilhável. No fim, vais ter um ZIP pronto, um repositório GitHub criado e um checklist objetivo para validar que tudo funciona na máquina de destino.

Phase 0: Preparar o pacote ⏱ 5 min

✓

Gerar o ZIP distribuível

Setup

Cria um arquivo limpo do projeto para partilha. O ZIP deve excluir a virtualenv e outros artefactos locais para não crescer sem necessidade.

Do

PowerShell 
powershell -ExecutionPolicy Bypass -File scripts/package_release.ps1

Pattern: um único script reprodutível que cria sempre o mesmo pacote em dist/.

Verify

Verificação inicial. Depois de correr o comando, verifica a pasta dist/. Deve aparecer um ficheiro ZIP novo. Isso confirma que o empacotamento está ativo.

Inclui código, dados, docs HTML, workflow CI, módulo de learning e scripts. Exclui .venv, pasta .git e ficheiros temporários.

💡

Phase 1: Criar e ligar o repositório ⏱ 5 min

✓

Criar o repositório vazio no GitHub

Plan

Cria um repositório vazio com README desativado, porque o projeto já inclui estrutura e documentação próprias. O objetivo é evitar conflitos no primeiro push.

Do

GitHub CLI auth 
gh auth login
gh auth setup-git
GitHub CLI public repo 
gh repo create retail-analytics-learning-repo --public --description "Retail analytics study case with data pipeline, dbt, ML, Streamlit, and realistic Black Friday test case" --source . --remote origin --push
GitHub CLI private repo 
gh repo create retail-analytics-learning-repo --private --description "Retail analytics study case with data pipeline, dbt, ML, Streamlit, and realistic Black Friday test case" --source . --remote origin --push

Pattern: autenticação primeiro, depois nome do repositório, visibilidade, descrição, pasta local como source, remote origin e push imediato.

Antes de executar. Se preferires criar pelo browser, não adiciones README, .gitignore ou licença no ecrã inicial. O repositório deve nascer vazio.

Cria o repositório manualmente no GitHub, copia o URL HTTPS e depois usa:

git init
git add .
git commit -m "feat: initial retail analytics repository"
git branch -M main
git remote add origin [URL_DO_REPO]
git push -u origin main
✓

Confirmar o conteúdo publicado

Generate

Abre o repositório no GitHub e confirma se os ficheiros principais apareceram. O objetivo aqui é validar a publicação, não editar nada ainda.

Do

🔗
Abrir o hub principal
Confirma que o HTML central do projeto existe no repositório publicado

Verify

O output mostra. Na raiz do repositório devem existir index.html, package.json, requirements.txt, python/, streamlit/ e docs/.
Abrir repo no browser

✍

Phase 2: Instalar e validar ⏱ 5 min

✓

Instalar dependências na máquina de destino

Setup

Escolhe os comandos do sistema operativo da máquina onde o repositório vai correr. Todos os caminhos seguintes assumem que já entraste na pasta do projeto.

Do

Windows PowerShell 
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -r requirements.txt
macOS / Linux 
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt

Verify

Verificação. O último comando termina sem erro e a prompt passa a mostrar a virtualenv ativa.
✓

Correr pipeline, ML e dashboard

Generate

Agora executa o fluxo principal do projeto. O objetivo é gerar dados, treinar modelos, validar outputs e abrir a camada visual.

Do

Terminal 
python python/run_pipeline.py
python python/ml_retail.py
python python/validate_project.py --include-dbt
streamlit run streamlit/app.py

Pattern: pipeline base, treino ML, validação total, dashboard principal.

Para validar o cenário de campanha, corre também:

python python/generate_test_case_black_friday.py
python python/analyze_test_case_black_friday.py
streamlit run streamlit/case_black_friday_app.py
✓

Confirmar que o repositório está pronto para partilha

Finish

Faz a verificação final antes de convidar outras pessoas para usar o repositório. Esta etapa fecha o loop entre empacotamento, publicação e execução.

Do

  • Abre index.html e confirma os links principais.
  • Confirma que data/ml_outputs/ contém os quatro ficheiros esperados.
  • Confirma que o dashboard principal e o dashboard Black Friday abrem sem erro.
  • Confirma que a workflow CI existe em .github/workflows/cross-platform-validate.yml.

Verify

Estado final. Se estes quatro checks passarem, o repositório está pronto para onboarding, demo e colaboração técnica.
🏆

Projeto empacotado e repositório pronto!

O ZIP está preparado, o repositório pode ser criado e o fluxo técnico está documentado para a próxima pessoa.
