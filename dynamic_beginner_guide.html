3. What each tool does

SQLite

This is the simple application database. Imagine a small store app saving transactions.

Use it for: tables, inserts, primary keys, foreign keys, indexes, basic SQL.

DuckDB

This is the analysis engine. It is good for local analytics and larger SQL queries.

Use it for: aggregates, trends, rankings, window functions, analytical SQL.

dbt

This is the transformation layer. It turns loose SQL files into a structured project.

Use it for: staging models, marts, tests, docs, lineage.

PL/SQL

This is Oracle-style logic inside the database.

Use it for: procedures, functions, packages, triggers, centralized business logic.

4. Data model

We use a supermarket because it is easy to understand.

stores

Each row is one shop.

products

Each row is one product.

customers

Each row is one customer.

transactions

Each row is one purchase event.

transaction_items

Each row is one product inside one purchase.

fct_sales

Analytical fact table created from transaction data.

Important interview concept: the grain of transaction_items or fct_sales is one row per product in one transaction.
What does “grain” mean?

Grain means the exact level of detail of one row. If you do not know the grain, you can easily duplicate data or calculate metrics incorrectly.

6. SQL basics — explained slowly

SQL is how you ask questions to the database.

Question: which products sold most?

SELECT
    product_id,
    SUM(quantity) AS total_quantity_sold
FROM transaction_items
GROUP BY product_id
ORDER BY total_quantity_sold DESC;

What this does: it adds up quantities for each product and shows the biggest sellers first.

Question: revenue by store

SELECT
    t.store_id,
    SUM(i.quantity * i.unit_price) AS total_revenue
FROM transactions t
JOIN transaction_items i
    ON t.transaction_id = i.transaction_id
GROUP BY t.store_id
ORDER BY total_revenue DESC;

Why the join? because one table has the transaction header and the other has the product details.

Beginner explanation of SELECT / FROM / JOIN / GROUP BY
  • SELECT = what columns or calculations you want to see
  • FROM = which table you are reading
  • JOIN = combine tables
  • GROUP BY = summarize rows into groups

7. Advanced SQL — the part that makes you look stronger

This is where you move from “I know SQL” to “I can solve analytical problems.”

Top 2 customers per store

WITH customer_store_spend AS (
    SELECT
        t.store_id,
        t.customer_id,
        SUM(ti.quantity * ti.unit_price) AS total_spent
    FROM transactions t
    JOIN transaction_items ti
        ON t.transaction_id = ti.transaction_id
    GROUP BY t.store_id, t.customer_id
),
ranked AS (
    SELECT
        store_id,
        customer_id,
        total_spent,
        ROW_NUMBER() OVER (
            PARTITION BY store_id
            ORDER BY total_spent DESC
        ) AS rn
    FROM customer_store_spend
)
SELECT *
FROM ranked
WHERE rn <= 2;

Why this matters: it uses a CTE plus a window function.

Daily revenue with running total

SELECT
    t.store_id,
    t.transaction_date,
    SUM(ti.quantity * ti.unit_price) AS daily_revenue,
    SUM(SUM(ti.quantity * ti.unit_price)) OVER (
        PARTITION BY t.store_id
        ORDER BY t.transaction_date
    ) AS cumulative_revenue
FROM transactions t
JOIN transaction_items ti
    ON t.transaction_id = ti.transaction_id
GROUP BY t.store_id, t.transaction_date;

Why this matters: it shows trends over time using a window function.

Interview sentence: “I use window functions for ranking, trends, cumulative metrics and time-based comparisons.”

8. What dbt adds

Without dbt, you might just have random SQL files. With dbt, the project becomes organized and easier to trust.

Staging

This layer cleans raw data. Rename columns, cast types, standardize fields.

select
    cast(product_id as integer) as product_id,
    trim(product_name) as product_name,
    trim(category) as category,
    cast(price as numeric(10,2)) as price
from {{ ref('raw_products') }}

Mart / fact model

This layer creates business-ready models.

select
    ti.item_id,
    ti.transaction_id,
    t.store_id,
    t.customer_id,
    ti.product_id,
    t.transaction_date,
    ti.quantity,
    ti.unit_price,
    ti.quantity * ti.unit_price as total_price
from {{ ref('stg_transaction_items') }} ti
join {{ ref('stg_transactions') }} t
    on ti.transaction_id = t.transaction_id
Why is ref() important?

ref() tells dbt that one model depends on another. That gives you lineage, ordering, and safer project structure.

Why are dbt tests useful?

Because they catch problems like missing keys, duplicates, and broken relationships before people trust the analysis.

9. PL/SQL explained simply

PL/SQL is Oracle’s procedural language. Think of it as SQL plus logic: variables, conditions, procedures, functions, triggers.

Procedure

A procedure usually performs an action.

CREATE OR REPLACE PROCEDURE pr_calculate_store_revenue (
    p_store_id IN NUMBER
) AS
    v_total_revenue NUMBER(12,2);
BEGIN
    SELECT NVL(SUM(ti.quantity * ti.unit_price), 0)
    INTO v_total_revenue
    FROM transactions t
    JOIN transaction_items ti
        ON t.transaction_id = ti.transaction_id
    WHERE t.store_id = p_store_id;

    DBMS_OUTPUT.PUT_LINE('Store revenue: ' || v_total_revenue);
END;
/

Function

A function usually returns one value.

CREATE OR REPLACE FUNCTION fn_customer_transaction_count (
    p_customer_id IN NUMBER
) RETURN NUMBER AS
    v_transaction_count NUMBER;
BEGIN
    SELECT COUNT(*)
    INTO v_transaction_count
    FROM transactions
    WHERE customer_id = p_customer_id;

    RETURN v_transaction_count;
END;
/
Very simple difference: procedure vs function

Procedure = do something. Function = give something back.

What is a trigger?

A trigger runs automatically when something happens, for example when a product price changes.

Easy Bosch connection: “I used PL/SQL-style procedures, functions and triggers to practice database-side business logic and understand the Oracle mindset.”

12. Mini practice — test yourself

Question 1: What is the grain of the sales fact table?

One row per product inside one transaction.

Question 2: Why do we use DuckDB if we already have SQLite?

SQLite simulates operational storage. DuckDB is used as an analytical engine for easier and stronger analysis.

Question 3: Why use dbt?

dbt organizes SQL into layers, adds testing, creates documentation, and makes the transformation logic easier to maintain.

Question 4: What is the difference between a procedure and a function?

A procedure mainly performs an action. A function returns a value.

Question 5: What makes SQL “advanced” here?

Using CTEs, joins, aggregations, and especially window functions like ROW_NUMBER, RANK, LAG and cumulative sums.
